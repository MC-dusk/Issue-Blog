# 解决数据库自增值不更新的问题

使用 dbeaver 连接 mysql8 ，表启用了主键id自增，发现有时候虽然修改了表的AUTO_INCREMENT值，执行后也没报错，但再次进入面板查看还是旧值，而实际新增一行，却发现主键id确实变了（也可能没变）。

问题分为两部分：

# 1. 修改不生效

为了保证主键的唯一性，MySQL 有一个规则：

当使用ALTER TABLE ... AUTO_INCREMENT = N语句时，只有当 N 的值大于表中该自增列当前的最大值时，设置才会真正生效。

如果 N 小于或等于现有最大值，MySQL 会忽略指令，并将下一个自增值保持在 MAX(id) + 1，以避免新插入的数据与已有数据的主键发生冲突。

解决方法是先确定想要的新自增值，比如50，然后找出表中所有比50大的id，要么删了，要么改到小于50。又或者把新自增值设置为比表中最大id还大的数。

* 如何验证是否修改成功？

可以使用以下任一命令来查看表的创建语句，从而确认AUTO_INCREMENT的值是否已经更新。

方法一：使用SHOW CREATE TABLE

SHOW CREATE TABLE <tb_name>;

在返回的结果中，定位类似AUTO_INCREMENT=50的部分，如果和原来不一样表示修改成功。

方法二：查询information_schema

SELECT AUTO_INCREMENT 
FROM information_schema.tables 
WHERE table_name = '<tb_name>' 
AND table_schema = '<db_name>';

# 2. 修改后回显没变

问题不在于修改没生效，而在于缓存没更新。information_schema这个系统库为了性能，默认会缓存表的元数据（包括自增值），而不是每次都去读取实时数据。

DBeaver 正是通过查询information_schema.tables来获取并展示表的自增值等信息的。当这个信息源是“过期缓存”时，DBeaver 显示的数据自然也就不是最新的了。

在 MySQL 8.0 中，为了提升information_schema的查询效率，系统引入了一个名为information_schema_stats_expiry的参数。它定义了information_schema中表统计信息（包括AUTO_INCREMENT）的缓存有效期。默认值是 86400 秒，也就是 24 小时。

要让 DBeaver 立即显示正确的自增值，关键是刷新information_schema的缓存。有以下几种方法：

  1. 刷新特定表的统计信息

执行下面命令可以立即更新表在information_schema中的缓存信息。

ANALYZE TABLE <tb_name>;

执行后再在 DBeaver 中刷新一下表的信息，就能看到正确的自增值了。

  1. 全局关闭缓存

可以将information_schema_stats_expiry参数设置为 0，这样information_schema就会每次都实时查询存储引擎，不再使用缓存。

SET GLOBAL information_schema_stats_expiry = 0;

这样做虽然能让 DBeaver 的数据永远是最新的，但可能会因为频繁的元数据查询而对数据库性能造成一定影响，尤其是在元数据查询频繁的场景下。
